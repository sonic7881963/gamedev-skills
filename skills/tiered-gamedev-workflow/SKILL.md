# 分级游戏开发工作流（S/M/L + 复盘回路）

## 核心原则

流程的重量要匹配改动的风险，且**流程本身可迭代**：轻量化砍掉的只能是"文档仪式"，
永远不能是安全网（TDD、设计确认、全量验证、调试纪律）。

## 特性分级

| 级别 | 适用 | 流程 |
|---|---|---|
| **S** | 数据调整 / 单文件小改 | 三五行设计向用户确认 → 直接 TDD → 提交；规格写进 commit message |
| **M** | 单系统特性 | 一份短规格（决策+文件清单+测试清单）→ TDD → 提交；**不写全码实施计划** |
| **L** | 跨系统改造 | 完整流程：superpowers:brainstorming → 规格 → superpowers:writing-plans → superpowers:executing-plans |

拿不准 → **默认 M**（短规格是最便宜的保险）。决策集中一轮问完，设计确认后自主跑完。

## 升级判定（任一命中即升级）

- **S→M**：≥2 个需用户拍板的决策；无同类先例的新机制；预计动 >3 个文件；
  会改变既有测试的前提（平衡/时序）。
- **M→L**：跨系统改语义（改变既有系统间的契约，例如重定义"冷却"）；需重写 ≥2 个
  既有测试；动所有人依赖的公共地基（战斗状态机、编队指挥）；接口波及 ≥3 个调用方。
- **任何级→先停下问用户**：要碰用户未提交的工作；不可逆/对外操作；疑似需求理解偏差。

## 永不轻量的底线（所有级别）

1. TDD 红→绿（superpowers:test-driven-development）
2. 开工前一轮设计确认——"简单"改动最容易藏物理/数学陷阱（实例：射程翻倍这种
   两行数据改动，靠开工前 10 分钟分析才发现抛物线极限射程根本够不到新数值）
3. 提交前全量测试 ×1（并行跑，见 godot-headless-testing）
4. 意外失败必走 superpowers:systematic-debugging（探针定位，禁止猜改）

## 中途升级

判级是假设，证据可推翻。S 级执行中出现以下信号 → **停手补短规格升 M**：
红测暴露更深耦合 / 实际文件数超预估 / 任何既有测试意外翻红。

## 复盘回路（工作流自我纠错）

**触发**：逃逸 bug（提交后才发现）/ 返工（同处二改）/ 丢失破坏工作 / 同类判级错误
反复 / 用户指出流程问题 / 上游工具**或运行环境**（Claude Code harness、shell、
引擎）更新导致行为变化——环境换了，所有以旧环境为前提的规则要实测重审。

**动作**（修完问题后归因三问）：
1. 旧流程的哪一步本会拦住它？
2. 流程缺口还是执行失误？（执行失误不改规则）
3. 最小修订是什么？（加一条检查 / 调判级触发 / 回退某项轻量化）

**输出**：一句话修订建议报用户，同意后写入项目 CLAUDE.md 的 append-only 修订记录
（日期/事故/修订），并同步更新本 skills 库对应内容。

**防膨胀**：规则只能因事故而增、因连续多特性无价值而删——双向收敛。

## 红线自检

| 念头 | 现实 |
|---|---|
| "这次特殊，跳过设计确认直接写" | 简单改动藏陷阱的概率不比大改低 |
| "测试回头补" | 红→绿的顺序就是规格本身，补测只是证明"它是这样"而非"它该这样" |
| "判错级了但快做完了，硬撑完吧" | 中途升级只花 5 分钟，返工花几小时 |
| "这条规则碍事，悄悄不遵守" | 走复盘回路提议删除——规则要么生效要么正式废除 |
